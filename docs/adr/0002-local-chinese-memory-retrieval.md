# ADR-0002：本地中文记忆语义检索与混合召回

- 状态：Accepted
- 日期：2026-07-13

## 背景

M4–M9 的派生索引使用 64 维确定性哈希向量，只能验证离线索引、召回和恢复链路。中文短文本没有天然空格分词，“用户喜欢喝咖啡”和“你还记得我喜欢喝什么吗”通常不会落到相同哈希特征，因此不能作为产品级记忆召回。

权威记忆仍保存在每宠物独立的 `memory/ledger.sqlite3`。语义向量、全文关键词和融合分数都必须是可删除、可从账本重建的派生数据。

## 决策

1. 使用 `BAAI/bge-small-zh-v1.5`，通过应用自有 Python 3.13 sidecar 的 ONNX Runtime CPU 执行 INT8 模型；不增加翻译步骤或云端 embedding 请求。
2. ONNX、tokenizer 和模型只在首次实际检索、索引写入或主动重建时加载。启动主窗口、只做整理 provider 测试或打开兼容索引均不得加载 embedding session。
3. 每条向量使用 CLS pooling、L2 归一化、512 维 little-endian Float32 BLOB；数据库约束固定长度为 2048 字节。JSON 向量不再受支持。
4. v2 派生索引同时维护：
   - BGE 语义通道；
   - SQLite FTS5 `trigram` 关键词通道；
   - 标签、姓名/称呼、专有名词、数字和日期的结构化精确通道。
5. 各通道使用有界 rank fusion 排序，最终置信度由证据类型和通道一致性校准；不同通道的原始分数不直接相加，重要标记只提供有限加权。
6. 模型 ID、官方 revision、转换 revision、模型与 tokenizer SHA-256、量化方式、维度、dtype、pooling、归一化方式和 schema version 分别写入派生索引元数据。任一不匹配只允许 staging 重建，不原地改写旧索引。
7. staging 完成后必须校验记录数、章节分布、内容指纹、FTS 行数、BLOB 长度/有限值、元数据和 SQLite `quick_check`，关闭 Windows SQLite 句柄后才允许原子切换。
8. 对明确的记忆核对问题，sidecar 返回 `verified` 或 `unknown` 策略：只有精确证据无冲突，或语义分数和名次差达到评测阈值时才可成为硬约束；无答案或相互冲突时不返回记忆，并向聊天模型注入“承认不知道、禁止猜测”的 system 约束。

## 结果

- 中文语义改写不再依赖空格或翻译，关键词和结构化通道可保住姓名、数字、日期等 embedding 容易漏掉的事实。
- 1000 条 512 维向量的紧凑 BLOB 约为 2 MiB，索引仍可完全从账本重建。
- 首次模型加载可能超过单轮 1200 ms 召回 deadline；该轮按无记忆降级，取消不得切换未完成 staging。模型加载完成后由同一 sidecar 常驻复用。
- 当前开发 INT8 文件仅用于本地验证。生产分发必须在 M10 从官方 MIT 权重自行导出并量化，或完成转换资产的明确许可审计，同时锁定最终资产哈希和 NOTICE。
